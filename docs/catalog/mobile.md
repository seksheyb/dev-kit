# Mobile Engineer

#### `flutter-expert`

- **Why needed:** Flutter's whole value proposition is a single Dart codebase spanning iOS, Android, and beyond, but that only works if state management (Riverpod/Bloc), widget composition, and rebuild performance are handled correctly — generic guidance tends to reach for `setState` and `StatefulWidget` in ways that cause full-subtree rebuilds and jank.
- **What it does:** Drives a five-step workflow (setup, state, widgets, test, optimize) with tool-verified checkpoints — `flutter analyze`, `flutter test`, DevTools profiling — plus load-on-demand references for Riverpod, Bloc, GoRouter, widget patterns, and performance, and a troubleshooting table for common failures like jank or failed hot reload.
- **Why not vanilla Claude Code:** Without this skill, Claude Code has no built-in bias toward `ConsumerWidget` over `StatefulWidget`, no awareness that `ref.watch` requires `const` discipline to avoid rebuild storms, and no reflex to profile with DevTools before declaring a Flutter feature done.
- **When to use:** Building cross-platform Flutter 3+ apps, implementing Riverpod/Bloc state, setting up GoRouter navigation, building custom widgets/animations, or doing platform-specific implementations and performance optimization.
- **Then what:** Claude scaffolds routing and dependencies first, writes state as Riverpod providers or Bloc/Cubit classes with immutable state transitions, builds `const`-optimized widgets with proper list keys, and only calls a feature done after `flutter analyze` and `flutter test` come back clean.
- **Notes:** The skill explicitly calls out an anti-pattern table (app-wide `setState`, mutating state directly, building widgets inside `build()`) that maps to real jank bugs, not just style preferences.

#### `kotlin-specialist`

- **Why needed:** Kotlin's concurrency model (coroutines, Flow, structured concurrency) is easy to misuse — `GlobalScope.launch` and unhandled cancellation are common sources of leaks and crashes — and Kotlin Multiplatform adds the extra risk of platform-specific code leaking into shared `commonMain` modules.
- **What it does:** Runs a six-step workflow (analyze architecture, design sealed-class models, implement with coroutines/Flow, validate with detekt/ktlint, optimize, test with `runTest`/Turbine) and supplies references for coroutines/Flow, KMP expect/actual setup, Android Compose, Ktor servers, and DSL idioms, alongside worked examples of sealed-class state modeling and safe scope functions.
- **Why not vanilla Claude Code:** Vanilla Claude Code won't reliably flag `runBlocking` in production paths, `GlobalScope.launch`, or missing cancellation handling as bugs, and has no default preference for `sealed class` state modeling over loose enums/booleans or for `detekt`/`ktlint` as a required gate before proceeding.
- **When to use:** Building Kotlin apps needing coroutines, Kotlin Multiplatform (KMP) projects, Android apps with Jetpack Compose, Ktor servers, or type-safe DSL design — triggers include Flow API, suspend functions, sealed classes, and KMP.
- **Then what:** Claude models state as sealed classes consumed exhaustively via `when`, writes coroutine code with structured concurrency (never `GlobalScope`), verifies null safety and cancellation handling at an explicit checkpoint, and gates progress on a clean `detekt`/`ktlint` pass before moving to optimization and multiplatform tests.
- **Notes:** The skill treats coroutine cancellation verification as an explicit named checkpoint step (not just a lint rule), and flags "explicit API mode for libraries" as a MUST DO — a KMP-specific concern most generic Kotlin guidance skips.

#### `react-native-expert`

- **Why needed:** React Native/Expo apps fail in stack-specific ways — unoptimized `FlatList` renders, missing `SafeAreaView`/`KeyboardAvoidingView` handling, and native module mismatches between iOS and Android — that generic React knowledge doesn't cover because there's no browser DOM or single rendering target underneath.
- **What it does:** Provides a five-step workflow (setup with `expo doctor`, feature-based structure, implement with dual-platform verification, optimize with Flipper/DevTools, test on both platforms) plus an explicit error-recovery table for Metro bundler cache issues, iOS/Android build failures, and missing native modules, backed by references for Expo Router, platform handling, list optimization, and storage.
- **Why not vanilla Claude Code:** Generic React instincts (using `ScrollView` for lists, inline style objects, `setTimeout`-driven animation) actively hurt React Native performance and don't account for two separate native build systems, which this skill's MUST NOT DO list and platform-specific error recovery steps are built to prevent.
- **When to use:** Building a React Native or Expo mobile app, setting up tab/stack/drawer navigation, integrating native modules, improving scroll performance, handling SafeArea or keyboard input, or configuring Expo SDK projects.
- **Then what:** Claude defaults to `FlatList`/`SectionList` with `memo` + `useCallback` for renderers, wraps forms in `KeyboardAvoidingView` with platform-conditional `behavior`, splits platform-specific code via `Platform.select` or `.ios.tsx`/`.android.tsx` files, and verifies changes on both an iOS simulator and Android emulator before calling a feature done.
- **Notes:** The skill's error-recovery table is unusually concrete about tooling commands (`npx expo start --clear`, `npx expo install <module>`, checking `adb logcat`), reflecting how much RN debugging is about native build tooling rather than JS logic.

#### `swift-expert`

- **Why needed:** Modern Swift concurrency (async/await, actors, `Sendable`) and SwiftUI state management (`@Observable` vs. legacy `ObservableObject`) have shifted fast enough that default LLM output tends to reach for outdated patterns — completion-handler wrapping, manual `NSLock` synchronization, class-based inheritance — instead of the idiomatic Swift 5.9+ approach.
- **What it does:** Defines a five-step workflow (architecture analysis, protocol-first design, implementation, Instruments profiling, XCTest testing) with named validation checkpoints (`swift build`, `swift build -warnings-as-errors`, `swift test`) and paired DO/DON'T code examples covering async/await, `@Observable` view models, protocol-oriented design with associated types, and actor-based thread safety.
- **Why not vanilla Claude Code:** Without this skill Claude Code has no default bias toward actors over manual locking for thread safety, no reflex to check `Sendable` compliance or actor-isolation warnings, and no built-in preference for protocol-oriented composition over class inheritance — all Swift-specific idioms that plain code generation tends to get wrong.
- **When to use:** Building iOS/macOS/watchOS/tvOS applications with Swift 5.9+, implementing SwiftUI views and state, designing protocol-oriented architectures, handling async/await concurrency or actors, or working with server-side Swift (Vapor), UIKit integration, or Combine.
- **Then what:** Claude designs protocol-first APIs with associated types, isolates mutable shared state in actors instead of locks, uses `@Observable` for SwiftUI view models instead of `@Published`/`ObservableObject`, and runs `swift build -warnings-as-errors` to catch actor-isolation and `Sendable` issues before writing tests.
- **Notes:** Unlike the other three mobile skills, this one has no `related-skills` cross-references in its frontmatter, and its validation checkpoints are phrased as an explicit escalating sequence (`swift build` then `swift build -warnings-as-errors` then `swift test`) rather than a single gate.
