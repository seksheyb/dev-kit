# Core Discovery & Design — Role-Based Reference Catalog

## Role: Product / Requirements Owner

#### `brainstorming` (skill)
- **Why needed:** Teams jump straight to code on unexamined assumptions; "simple" requests are exactly where wasted work hides because nobody stopped to ask what's actually being built or why.
- **What it does:** Runs a collaborative dialogue that explores context, asks one question at a time, proposes 2-3 trade-off approaches, and gets section-by-section approval on a design — then hands off to `specify` to produce the actual written spec. Includes two alternate modes: a YC-office-hours diagnostic for pre-code product ideas and a go/no-go idea-validation mode that hunts for the idea's fatal flaw.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no hard gate preventing it from writing code the moment a request looks tractable — this skill enforces a `<HARD-GATE>` that blocks any implementation action until a design is presented and approved, for every project regardless of perceived simplicity.
- **When to use:** Frontmatter: "You MUST use this before any creative work - creating features, building components, adding functionality, or modifying behavior... Also use when the user has a product idea, asks 'is this worth building', wants office-hours-style pressure testing, or needs a go/no-go validation."
- **Then what:** Once every design section is approved, hands off to `specify` with the approved design as context, so `specify` doesn't re-ask what this conversation already settled. `specify` owns spec production, its own quality checklist, and its own user-approval gate from that point on.
- **Notes:** Terminal state is explicitly "an approved design handed to `specify`" — the skill instructs itself never to write the spec file, run spec-level quality checks, or invoke planning directly; that's `specify`'s and `writing-plans`'s job respectively. (Prior to a Stage 0/1 asset-overlap audit, this skill wrote and self-reviewed its own separate spec file under a different naming convention than `specify`'s — a duplicate "idea → spec → plan" path that bypassed `specify`, the Clarification Pass, and `spec-review-cpo` entirely. Steps 5-8 of the old checklist were removed and folded into `specify` instead.)

#### `constitution` (skill)
- **Why needed:** Without a durable, versioned statement of project principles, every downstream spec/plan/task can silently drift from what the team actually agreed to, and there's no single place to check a proposed change against non-negotiable rules.
- **What it does:** Initializes or amends a semantically-versioned `docs/constitution.md` from a template, resolves placeholder tokens into concrete principles, and propagates the change across dependent SDD artifacts (spec/plan/tasks templates), emitting a Sync Impact Report of what changed.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no persistent, versioned principles document that later commands treat as binding authority — without this skill, "project rules" live only in scattered chat history and get reinterpreted each session.
- **When to use:** Frontmatter: "Use when the user says 'create a constitution', 'project principles', 'update the constitution', 'set project ground rules', 'governance rules', or when starting spec-driven development on a project that has no constitution yet."
- **Then what:** The constitution file is written/overwritten with a version bump and Sync Impact Report; the skill suggests running `specify` next now that a governing constitution exists.
- **Notes:** Downstream skills treat it as non-negotiable authority — `analyze` flags any conflict with a MUST principle as CRITICAL, and `converge` emits constitution-violation remediation tasks first.

#### `specify` (skill)
- **Why needed:** Feature ideas arrive as vague one-liners or half-formed descriptions; without a forcing function, "build me a chat feature" becomes ambiguous requirements that surface as rework during implementation. A spec that "looks complete" can also still hide ambiguous data models, unhandled edge cases, or unresolved non-functional targets that surface expensively during implementation instead of cheaply during planning — that's a different, semantic kind of gap than structural completeness catches.
- **What it does:** Converts a feature description into a structured `spec.md` (WHAT/WHY only, never HOW) via direct generation with up to 3 targeted clarification questions, or via a five-phase interrogation for genuinely vague requests, interviewing from a PM Hat (value/goals) and Dev Hat (feasibility/security/edge cases). Functional requirements use EARS format where conditional (`references/ears-syntax.md`); non-functional requirements need a real, testable target. Self-validates against a generated structural quality checklist, then closes with an inline **Clarification Pass**: a structured ambiguity scan across a fixed taxonomy (functional scope, data model, UX flow, non-functional attributes, integrations, edge cases, terminology, completion signals) that asks up to 5 highest-impact questions one at a time and writes each accepted answer back into the spec's `## Clarifications` section.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standard spec template, clarification-question budget, or EARS discipline — it either asks unlimited scattered questions or guesses silently, and clarifying questions in a normal session are ad hoc and can spiral without limit or get skipped entirely. This skill caps generation-time clarifications at 3 (prioritized by scope/security/UX impact), enforces a hard 5-question cap for the post-generation ambiguity scan, and integrates answers atomically into the spec file after each one.
- **When to use:** Frontmatter: "Use when the user says 'write a spec', 'specify this feature', 'create a feature specification', 'turn this idea into a spec', 'I want to build...', 'clarify the spec', 'resolve ambiguities', 'de-risk the spec', or provides a feature description that needs to become a formal, testable specification before planning."
- **Then what:** A numbered feature directory (`docs/specs/NNN-feature-name/spec.md` plus a requirements checklist) is written, the Clarification Pass runs and updates it in place with a dated Clarifications log plus a before/after checklist re-validation, and then — a hard gate — the user is asked to explicitly approve the spec before the skill will hand off to `spec-review-cpo` or planning.
- **Notes:** Loads the project constitution (if one exists) to ensure the spec doesn't conflict with governing principles. Before drafting, checks whether the request actually describes multiple independent subsystems and decomposes into sub-projects first rather than drafting one oversized spec. The Clarification Pass is independently re-invocable against an already-written spec ("clarify the spec") without redoing generation. Deliberately does **not** produce an implementation TODO checklist — that's `writing-plans`/`planner`'s job, not this skill's; a spec that ships a task list is scope creep into planning territory. Absorbed the former standalone `clarify` and `feature-forge` skills, and (after a further audit) the scope-decomposition check, internal-consistency check, and user-approval gate formerly duplicated inside `brainstorming`'s now-removed steps 5-8 — see [`gwd-pipeline-on-devkit.md`](../gwd-pipeline-on-devkit.md) for why.

#### `analyze` (skill)
- **Why needed:** Once a spec and its phase `PLAN.md` both exist, they can quietly diverge — a requirement with zero tasks, a plan decision no task implements, or terminology drift — and nobody catches it until implementation stalls on a gap.
- **What it does:** A strictly read-only audit that builds semantic models of the spec and the phase `PLAN.md` (tasks embedded as `<task>` blocks) plus the constitution, then runs six detection passes (duplication, ambiguity, underspecification, constitution alignment, coverage gaps, inconsistency) and emits a severity-graded Markdown report with a requirement-to-task coverage table.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no cross-artifact consistency checker — a human has to manually cross-reference spec and plan by eye; this skill produces stable, rerunnable finding IDs and a hard cap of 50 findings so the review stays token-efficient and deterministic.
- **When to use:** Frontmatter: "Use when the user says 'analyze the spec', 'analyze the plan', 'check consistency', 'audit the artifacts', 'find gaps between spec and plan', or before a phase's plan is executed."
- **Then what:** A report is printed (no files modified) with CRITICAL/HIGH/MEDIUM/LOW findings and next-action suggestions; the skill then offers to draft remediation edits, which the user must explicitly approve.
- **Notes:** Reads this pipeline's single-file plan convention (`PLAN.md` with embedded `<task>` blocks) — the same convention Stage 11's `converge` now reads and appends to, a Stage 7/11 audit having retargeted `converge` off the Spec-Kit `spec.md`/`plan.md`/`tasks.md` triad, which nothing in dev-kit produces. Any conflict with a constitution MUST principle is automatically CRITICAL and can only be resolved by updating the spec/plan or by a separate, explicit `constitution` amendment — never by silently reinterpreting the principle here.

#### `assumption-mapping` (skill)
- **Why needed:** Every product idea rests on unstated assumptions about value, usability, business viability, and feasibility; without surfacing and ranking them, engineering effort gets spent building on the assumption most likely to be wrong.
- **What it does:** Extracts assumptions across the 4 VUBF risk categories (Value/Usability/Business-viability/Feasibility), scores each by importance × evidence-strength on a prioritization grid, and designs the cheapest experiment to test the top 3-5 riskiest ones first.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standing risk taxonomy to sort product assumptions into — without it, "what could go wrong" discussions are unstructured and tend to fixate on whichever risk was mentioned last, not the one with the worst importance/evidence ratio.
- **When to use:** Frontmatter: "Use when identifying and prioritizing risky assumptions in a product idea, feature, or strategy before committing engineering effort... Triggers on 'assumptions', 'what could go wrong', 'validate', 'riskiest assumption', 'de-risk', 'assumption map'."
- **Then what:** An assumption table (Assumption | Category | Importance | Evidence | Priority) plus per-assumption experiment design and validate/invalidate decision rules is delivered to the user.
- **Notes:** Related-skills metadata ties it to `backlog-grooming` — validated assumptions naturally feed into what gets prioritized into the backlog next.

#### `spec-miner` (skill)
- **Why needed:** Inherited or undocumented codebases have no reliable requirements source — the only ground truth is the code itself, and guessing at intent from memory produces a spec that doesn't match what's actually running.
- **What it does:** Explores a codebase in two passes (Arch Hat for system architecture/data flow, QA Hat for observable behavior/edge cases) using Glob/Grep/Read, then documents observed requirements in EARS format (Ubiquitous/Event-driven/State-driven/Optional) with file-path citations and a dedicated uncertainties section.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no enforced evidence discipline for reverse-engineering — it can hallucinate plausible-sounding requirements from a quick skim; this skill's MUST-NOT-DO list explicitly forbids assumptions without code evidence and requires a validation checkpoint before writing documentation.
- **When to use:** Frontmatter: "Use when working with legacy or undocumented systems, inherited projects, or old codebases with no documentation... Trigger phrases: reverse engineer, old codebase, no docs, no documentation, figure out how this works, inherited project, legacy analysis, code archaeology, undocumented features."
- **Then what:** A `specs/{project_name}_reverse_spec.md` is produced with stack/structure, EARS-format requirements, non-functional observations, inferred acceptance criteria, and open questions.
- **Notes:** The `gate-reverse-engineer` agent explicitly follows this skill's methodology as its extraction technique when recovering a Legacy SDD/PRD from an existing codebase.

#### `backlog-grooming` (skill)
- **Why needed:** Backlogs left ungroomed accumulate zombie stories, vague acceptance criteria, and priorities that no longer reflect current strategy — an unhealthy backlog quietly buries a team's throughput.
- **What it does:** Runs a structured grooming session methodology (hygiene → story refinement → priority review), applies estimation methods (story points, t-shirt sizing, planning poker), enforces a Definition-of-Ready checklist, and covers the surrounding agile ceremonies (planning, standup, review, retro, impediment removal).
- **Why not vanilla Claude Code:** Vanilla Claude Code has no opinionated backlog-health standard to check against — it will happily discuss "what should we build next" without ever flagging that half the backlog is stale or that a story lacks testable acceptance criteria.
- **When to use:** Frontmatter: "Use when grooming, refining, or cleaning up a product backlog — estimating stories, writing acceptance criteria, pruning zombie items, and preparing sprint-ready work... Triggers on 'groom backlog', 'backlog refinement', 'backlog grooming', 'clean up backlog', 'refine stories', 'sprint refinement', 'backlog management'."
- **Then what:** Delivers a groomed-backlog assessment: which stories are sprint-ready vs. need more work, items recommended for archival, and a refinement agenda for the next session.
- **Notes:** Related-skills metadata links it to `assumption-mapping` (validated assumptions become backlog items) and `feature-forge`.

#### `converge` (skill)
- **Why needed:** After an implementation pass, it's easy to lose track of exactly which spec requirements, plan decisions, and existing plan tasks are actually satisfied by the current code versus only partially done — without a systematic check, gaps ship silently.
- **What it does:** Treats `spec.md` and the phase's `PLAN.md`(s) — tasks embedded as `<task>` blocks, plus the constitution and, when present, the phase's `VERIFICATION.md` gaps as pre-confirmed evidence — as the sole source of intent, assesses the current codebase against that intent, classifies gaps as missing/partial/contradicts/unrequested with severity, and appends each as a new traceable `<task>` block under a `## Phase N: Convergence` header at the end of the relevant `PLAN.md` — never rewriting existing content.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no append-only, traceable gap-closure mechanism — asking "what's left to build" in a normal session produces a prose answer with no persistent, source-referenced task list; this skill enforces a strict append contract and constitution-violation-first ordering.
- **When to use:** Frontmatter: "Use when the user says 'converge', 'diff the code against the spec', 'what's left to build', 'gap analysis vs spec', 'check if the implementation matches the spec', or after an implementation pass to find remaining work."
- **Then what:** Either the phase's `PLAN.md` gains a new Convergence phase of remediation `<task>` blocks ready for an implementation pass (`sprint-execution`), or (if fully satisfied) every `PLAN.md` is left byte-for-byte unchanged and a "✅ Converged" report is issued with a recommendation to proceed to `verify` (if not already run) or review/PR.
- **Notes:** Explicitly NOT a diff tool — it assesses present-state code against artifacts, with no git/branch history involved, and is designed to be safely re-run (a later run should find fewer or no remaining items). Retargeted by a Stage 7/11 audit off the Spec-Kit `spec.md`/`plan.md`/`tasks.md` triad (nothing in dev-kit produces `plan.md`/`tasks.md`) onto this pipeline's `spec.md` + embedded-task `PLAN.md` convention; it is the **remediation compiler** at Stage 11, distinct from `verifier`'s goal-backward **verdict** (see [`core-build-and-ship.md`](core-build-and-ship.md)).

#### `the-fool` (skill)
- **Why needed:** Plans and proposals reviewed only by their own author accumulate blind spots — nobody internally is incentivized to argue the strongest opposing case or find the failure mode the team doesn't want to see.
- **What it does:** Steelmans the user's position, then applies one of 5 structured reasoning modes chosen via `AskUserQuestion` (Socratic questioning, Hegelian dialectic, pre-mortem, red-teaming, or falsificationism/evidence-audit) to produce 3-5 concrete challenges, followed by a synthesis that strengthens the original position.
- **Why not vanilla Claude Code:** Vanilla Claude Code left to critique its own or the user's plan tends toward polite hedging ("that's an interesting approach") rather than committed adversarial reasoning — this skill explicitly forbids strawmanning, forbids nihilistic pile-on, and requires driving to a synthesis rather than leaving just objections.
- **When to use:** Frontmatter: "Use when challenging ideas, plans, decisions, or proposals using structured critical reasoning. Invoke to play devil's advocate, run a pre-mortem, red team, or audit evidence and assumptions."
- **Then what:** Produces a steelmanned thesis, the 3-5 strongest challenges, space for the user to respond, a strengthened synthesis, and an offer of a second pass with a different mode.
- **Notes:** Related-skills metadata ties it to `architecture-designer` and `specify` — it's meant to stress-test a design or spec those skills already produced, not to originate one.

#### `spec-review-cpo` (skill)
- **Why needed:** Real scope/strategy judgment (is this the right problem, is prioritization evidenced, should we expand or cut) needs to happen at the spec stage — before a plan, and often architecture, is written, when reversing a bad scope call is far more expensive. This is the pipeline's single product/strategy gate; there is no founder-mode plan-stage re-review to fall back on.
- **What it does:** Commits to one of four scope postures (Expansion/Selective-Expansion/Hold/Reduction) sized for a bare spec, runs a premise challenge and existing-solution-leverage check — escalating to a first-principles decomposition (list every assumption, challenge each, rebuild from fundamental truths) when the framing itself looks inherited-by-convention rather than reasoned — produces 2-3 competing solution *framings* (never implementation approaches — stays inside the spec's WHAT/WHY boundary), and scores prioritization via RICE/Kano/JTBD/North Star against the `US-xxx` Theme→Pillar hierarchy when present. Writes a locked **Scope Decision Record** into the spec's `## CPO Review` section.
- **Why not vanilla Claude Code:** Vanilla Claude Code asked to sanity-check a spec's ambition level has no forced posture commitment or prioritization framework — it tends to accept the stated scope at face value rather than actively arguing for expansion, reduction, or a different problem framing entirely, and has no mechanism to make that judgment binding on later work.
- **When to use:** Use when reviewing a spec for strategy, scope, and product value — after `specify` (including its clarification pass), before `writing-plans`/`planner` — or when asked for a "CPO review", "product review of the spec", "is this worth building", "scope call before we plan".
- **Then what:** Appends a `## CPO Review` section (same in-place pattern as `specify`'s Clarification Pass uses for `## Clarifications`) to `spec.md` with posture, premise verdict, alternatives, prioritization scores, and a milestone split — plus a **LOCK** statement that the scope decision doesn't get re-litigated downstream. Anything descoped to a later milestone is also written to `docs/BACKLOG.md` (created if missing) as a durable, ID-tracked entry — never left as prose only.
- **Notes:** `docs/BACKLOG.md` shares `backlog-grooming`'s Now/Next/Later/Icebox/Won't-Do taxonomy and carries `US-xxx`/Pillar IDs forward when the story-bank hierarchy is in use, so `backlog-grooming` can groom it with the same discipline it already applies to any other backlog. A new milestone's `specify` pass reads this file as its primary requirements input — a milestone is treated exactly like a new project, run end-to-end through the whole pipeline again, seeded from whatever this skill previously deferred. `market-researcher`'s sourced data and `assumption-mapping`'s VUBF table are consumed as input evidence here, not re-derived; `the-fool`'s adversarial modes remain available as an optional extra pressure-test but are no longer required by default now that this lens's own posture is inherently adversarial. This skill's Scope Decision Record is the pipeline's only scope/strategy gate — there is no plan-stage founder review (no `plan-review-ceo`). Its Stage 2 architecture counterpart is `sdd-review-cto`, which does the same thing for the SDD (technical soundness) that this skill does for the spec (product scope).

#### `market-researcher` (agent)
- **Why needed:** Product decisions made without sourced market/competitive/trend intelligence are guesses dressed up as strategy — someone has to actually triangulate sizing, competitor positioning, or emerging signals before a roadmap commits to a direction.
- **What it does:** Runs one of three research lenses (market-sizing, competitive, or trends) or a combined lighter scan, applies source triangulation (2+ independent sources per key claim) and honest confidence tagging, and writes a sourced `MARKET.md` with strategic implications mapped to each finding.
- **Why not vanilla Claude Code:** Vanilla Claude Code doing ad hoc market research in a normal chat has no discipline forcing it to triangulate sources or flag single-source claims as LOW confidence — this agent's methodology makes every figure carry a source and date, and requires searching for disconfirming evidence before writing conclusions.
- **When to use:** Dispatched by the orchestrator/pipeline with a `focus` argument (`market-sizing | competitive | trends`); omitting it runs a combined scan flagging which lens deserves a dedicated follow-up.
- **Then what:** `.planning/research/MARKET.md` is written with an executive summary, focus-specific findings, strategic implications, and a gaps/follow-ups section; a one-paragraph summary with the single most decision-relevant recommendation returns to the orchestrator.
- **Notes:** Combines three analyst disciplines (market analyst, competitive intelligence, trend forecaster) behind one dispatch argument rather than requiring three separate agents.

#### `domain-researcher` (agent)
- **Why needed:** When building an AI system for a specialized domain (legal, healthcare, finance), "the AI should be accurate" is meaningless — what a domain expert actually checks for (citation precision, regulatory constraints, known failure modes) has to be researched before anyone can write a real evaluation rubric.
- **What it does:** Extracts the domain signal from upstream phase artifacts, runs 2-3 targeted searches for practitioner evaluation criteria and production failure modes, and synthesizes 3-5 domain-specific rubric ingredients (Dimension/Good/Bad/Stakes/Source) plus regulatory context and domain-expert roles, writing them into AI-SPEC.md Section 1b.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no step dedicated to researching the business domain separately from the technical framework — without this agent, eval criteria default to generic AI jargon ("accuracy", "helpfulness") instead of the practitioner language a domain expert would actually use to judge output.
- **When to use:** Dispatched by the `/gsd:ai-integration-phase` orchestrator when a phase involves building an AI system that needs a domain-grounded evaluation contract.
- **Then what:** Section 1b of AI-SPEC.md is written with domain context, rubric ingredients, known failure modes, regulatory constraints, and expert roles — consumed downstream by the eval-planner that turns it into measurable rubrics.
- **Notes:** Explicitly forbidden from fabricating criteria — if the domain is genuinely unclear it writes a minimal section flagging what needs clarification with domain experts, rather than inventing plausible-sounding rubric ingredients.

#### `project-researcher` (agent)
- **Why needed:** Before a roadmap can be created for a new project, someone has to investigate what the ecosystem for this kind of product actually looks like (standard stack, table-stakes features, architecture patterns, common pitfalls) — skipping this produces a roadmap built on guesses rather than evidence.
- **What it does:** Investigates one of three modes (Ecosystem/Feasibility/Comparison), applies a strict tool-priority hierarchy (Context7 → official docs → WebSearch, cross-verified) and four search lenses (source-type, precision-retrieval, literature-rigor, synthesis), then writes STACK/FEATURES/ARCHITECTURE/PITFALLS (plus COMPARISON or FEASIBILITY) files to `.planning/research/`. Never writes `SUMMARY.md` — that's `research-synthesizer`'s exclusive output, so a domain-scoped researcher doesn't race against (or duplicate, with drift, via its own template) the synthesis step.
- **Why not vanilla Claude Code:** Vanilla Claude Code's training data is 6-18 months stale and it will state library capabilities with unwarranted confidence — this agent treats training data as hypothesis, requires verification before asserting, and explicitly tags every finding HIGH/MEDIUM/LOW confidence rather than presenting a guess as fact.
- **When to use:** Dispatched by the `/gsd:new-project` or `/gsd:new-milestone` orchestrators (Phase 6: Research) before roadmap creation.
- **Then what:** Files are written directly to `.planning/research/`, feeding phase-structure suggestions, ordering rationale, and research flags into the roadmapper via `research-synthesizer`'s `SUMMARY.md`; the agent does not commit — that's the `research-synthesizer`'s job.
- **Notes:** Explicitly does NOT commit its own output — it's designed to run in parallel with three sibling researchers, all of whose files the `research-synthesizer` agent later reads, synthesizes into the one `SUMMARY.md`, and commits together.

#### `advisor-researcher` (agent)
- **Why needed:** Mid-planning "gray area" decisions (which library, which pattern) need a quick, honest options comparison rather than either a full research report or an unresearched guess — someone has to produce exactly one well-reasoned table per open question.
- **What it does:** Researches a single assigned gray-area decision using Context7/WebFetch/WebSearch in priority order, and returns a strict 5-column comparison table (Option/Pros/Cons/Complexity/Recommendation) sized 2-5 options by a calibration tier, plus one grounded rationale paragraph — never ranking a single winner, always conditional recommendations.
- **Why not vanilla Claude Code:** Vanilla Claude Code asked "which library should I use" in the middle of a planning session tends to sprawl into an unbounded comparison or a confident single pick — this agent is scoped to exactly one gray area, forbidden from time estimates in the Complexity column, and forbidden from declaring a single winner.
- **When to use:** Dispatched by the orchestrator/pipeline in "advisor mode" for one specific unresolved decision surfaced during discuss-phase.
- **Then what:** Returns structured markdown (table + rationale) for the main agent to synthesize alongside other advisor-researcher outputs — it never presents directly to the user.
- **Notes:** Complexity is defined strictly as impact-surface-plus-risk (e.g. "3 files, new dep — Risk: memory, scroll state"), never a time estimate — a deliberate anti-pattern guard against sprint-estimate creep into research output.

#### `research-synthesizer` (agent)
- **Why needed:** Four parallel researcher outputs (stack, features, architecture, pitfalls) are individually useful but not roadmap-ready until someone integrates them into one coherent narrative with explicit phase-structure implications — otherwise the roadmapper has to re-derive synthesis itself from four separate files.
- **What it does:** Reads all 4 research files, writes a 2-3 paragraph executive summary, extracts key findings per file, derives suggested phase structure with rationale and research flags (which phases need deeper research during planning), assesses confidence honestly, and commits all research files together.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no dedicated synthesis step that turns four separate research documents into one opinionated roadmap-ready summary — without it, the roadmapper agent would need to read and re-synthesize four raw files itself, duplicating work and diluting the "be opinionated, not wishy-washy" discipline this agent is built for.
- **When to use:** Dispatched by the orchestrator/pipeline after the 4 parallel researcher agents (STACK/FEATURES/ARCHITECTURE/PITFALLS) complete.
- **Then what:** Writes and commits `.planning/research/SUMMARY.md` (the only of the 5 research files it commits — for all of them, since researchers don't commit their own), then reports readiness for the requirements/roadmap phase.
- **Notes:** It is the one agent in the research pipeline responsible for the git commit of all research files — the four researcher agents deliberately write but never commit.

#### `assumptions-analyzer` (agent)
- **Why needed:** Before locking a phase's technical approach, someone needs to deeply read the actual codebase (not just skim it) and surface which decisions are well-evidenced versus genuinely unclear — guessing at "the codebase probably does X" produces plans built on assumptions nobody checked.
- **What it does:** Reads the ROADMAP phase description and prior CONTEXT.md files, searches for and reads 5-15 relevant source files, then returns structured assumptions (2-5 areas depending on calibration tier) each with a file-path-cited "why this way," a concrete "if wrong" consequence, and an honest Confident/Likely/Unclear confidence label — plus a "Needs External Research" list for anything codebase analysis alone can't resolve.
- **Why not vanilla Claude Code:** Vanilla Claude Code will often state an inferred codebase behavior as fact without citing where it read it — this agent's rules mandate every assumption cite at least one file path and forbid inflating confidence when evidence is thin, and it explicitly has no web/external tools, keeping it strictly grounded in what the code actually shows.
- **When to use:** Dispatched by the orchestrator/pipeline in "assumptions mode" during discuss-phase, for a single phase.
- **Then what:** Returns structured markdown (not presented directly to the user) for the main discuss-phase workflow to present and get confirmation on, before those assumptions harden into locked decisions.
- **Notes:** Explicitly forbidden from suggesting scope expansion or including implementation details — those are the planner's job; this agent's sole output is calibrated, evidenced assumptions.

## Role: Architect / Planner

#### `architecture-designer` (skill)
- **Why needed:** New systems or significant architectural changes made without documented trade-offs accumulate technology choices nobody can later explain or defend — someone eventually asks "why did we pick this?" and there's no record.
- **What it does:** Walks through a 5-step workflow (understand requirements → identify patterns → design with explicit trade-offs and a diagram → document via ADRs → review with stakeholders) and outputs a requirements summary, a Mermaid architecture diagram, ADR-formatted key decisions, technology recommendations, and risk/mitigation notes.
- **Why not vanilla Claude Code:** Vanilla Claude Code will describe an architecture in prose without forcing itself to produce a durable, alternatives-considered ADR — this skill's MUST-DO list requires documenting every significant decision with an ADR and evaluating trade-offs, not just benefits, before finalizing.
- **When to use:** Frontmatter: "Use when designing new high-level system architecture, reviewing existing designs, or making architectural decisions... distinct from code-level design patterns or database-only design tasks."
- **Then what:** Produces a diagram plus a set of ADR documents that `sdd-review-cto` gates at Stage 2, and that subsequent phase plans and their Stage 7 `plan-review-eng` lens check against.
- **Notes:** Related-skills metadata links it to `fullstack-guardian`, `devops-engineer`, and `secure-code-guardian` — it's meant to hand off into implementation-focused skills once the architecture is locked.

#### `sdd-review-cto` (skill)
- **Why needed:** A system design (SDD + ADRs) can look complete while resting on an unjustified technology bet, an ADR that records a decision but never its alternatives, or a pattern that won't meet a stated non-functional requirement — and once phases build on it, reversing a bad architecture call is expensive. Before this skill, the pipeline claimed `plan-review-eng` covered the SDD, but that lens is written for a task-bearing `PLAN.md` (codepath tracing, test-coverage diagrams) and can't do a test-coverage pass on a design doc.
- **What it does:** The Stage 2 counterpart to `spec-review-cpo` (CPO settles *what* to build; CTO settles *whether the architecture is sound to build on*). Commits to one of four technical postures (Soundness-Hold/Simplify/Re-architect/De-risk), runs an architecture premise challenge (requirements traced to design elements, innovation-token audit, minimum-viable-architecture check), reviews every ADR for context/alternatives/trade-offs/reversibility, assesses scalability posture and technical-debt trajectory, and writes a locked **Architecture Decision Record** into the SDD's `## CTO Review` section.
- **Why not vanilla Claude Code:** Vanilla Claude Code asked "is this architecture sound?" has no forced posture, no ADR-quality rubric, and no innovation-token discipline — it accepts the stated design at face value; this skill forces a committed posture, treats an unmet hard requirement or a one-way-door decision made with no alternatives as an automatic BLOCKER, and makes the verdict binding on downstream plans.
- **When to use:** Use when reviewing an SDD/architecture for soundness before the roadmap — or when asked for a "CTO review", "architecture review", "is this design sound", "review the SDD before we plan".
- **Then what:** Appends a `## CTO Review` section (posture, soundness verdict, innovation tokens, ADR gaps, unmet requirements, scalability posture, debt trajectory, reversibility) with a LOCK line, and a verdict of SOUND / SOUND-WITH-CHANGES / UNSOUND.
- **Notes:** Defers security depth to `cso` rather than re-running a threat model — `cso` is not concurrent with this review (a Stage 8–13 audit moved it off Stage 2 to Stage 0/13, since every one of its phases needs code that doesn't exist yet at Stage 2 of a first milestone), so this skill weighs whatever `.security-reports/` entry already exists rather than a fresh pass. Does **not** review any phase's `PLAN.md` — that's `plan-review-eng`'s Stage 7 job against a different artifact. Replaced the removed `plan-review-ceo` in a net-zero skill swap during a Stage 2/7 audit.

#### `writing-plans` (skill)
- **Why needed:** A spec describes what to build, but an implementer with zero codebase context still needs exact file paths, exact interfaces, and exact verification steps — without that level of specificity, plans become "figure it out yourself" documents that produce inconsistent, unreviewable diffs. It is also the single source of truth for the `PLAN.md` task format, so the interactive path and the pipeline `planner` don't diverge into two conventions.
- **What it does:** Defines the canonical plan-authoring methodology and `<task>` format — file-structure mapping, task right-sizing, the `<task>` block (`<files>`/`<interfaces>`/`<action>`/`<verify>`/`<done>`/`<complexity_signals>`), a no-placeholders rule, goal-backward must-haves, and a self-review pass. Used two ways: **standalone** for interactive planning, and **invoked by the `planner` agent** to author each pipeline plan. Both emit the same task format, so both flow through `gate-plan-review` and `sprint-execution` unchanged.
- **Why not vanilla Claude Code:** Vanilla Claude Code asked to "write a plan" tends to produce prose descriptions of what should happen rather than a literal, executable prompt — this skill's "No Placeholders" section forbids vague action items, mandates `<complexity_signals>` on every task, and requires a self-review pass checking spec coverage, placeholder scan, type consistency, and signal honesty before the plan is done.
- **When to use:** Frontmatter: "Use when you have a spec or requirements for a multi-step task, before touching code."
- **Then what:** Standalone, a plan is saved to `docs/plans/YYYY-MM-DD-<feature-name>.md`; in the pipeline the `planner` supplies the `.planning/phases/.../<NN>-PLAN.md` path and frontmatter. Either way the user is offered parallel (wave-based, worktree-isolated) or sequential execution, both funneling into `sprint-execution`.
- **Notes:** Explicitly requires the `sprint-execution` skill for execution — the plan header states this as a "REQUIRED SUB-SKILL," treating sequential execution as just a sprint where every wave has one track. Paired with the `planner` agent (method-skill + dispatch-agent), the same pairing pattern as `systematic-debugging`↔`debugger` and the plan-review lenses↔`plan-reviewer`.

#### `sprint-execution` (skill)
- **Why needed:** Executing a multi-task plan by hand (or via ad hoc subagent calls) risks race conditions on shared files, lost context after compaction, and reviewers who never actually see the real diff — a written plan alone doesn't guarantee safe, parallel, verifiable execution.
- **What it does:** Dispatches one subagent per track per wave into isolated git worktrees, enforces a mandatory `git reset --hard` base-sync per track, requires TDD-first task contracts, handles per-task file handoffs (briefs in, reports out) to keep orchestrator context thin, and runs two-stage review gates (per-wave and final whole-branch) plus a durable progress ledger for compaction-proof recovery.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no built-in wave-gating or worktree-isolation discipline for parallel subagent work — without this skill, simultaneous subagents can race on the same working tree, and a compacted session has nothing but its own (unreliable) memory to know what's already done; this skill anchors truth in the ledger and `git log` instead.
- **When to use:** Frontmatter: "Use when executing a written sprint/implementation plan with multiple tasks or parallel tracks... Trigger when asked to 'execute the plan', 'run the sprint', 'dispatch the waves', or when a plan file with a parallel execution map exists."
- **Then what:** Each wave's worktrees are merged and state files updated before the next wave starts; once the final whole-branch review comes back clean, control hands off to the `finishing-a-development-branch` skill for the merge/PR/keep/discard decision.
- **Notes:** Explicitly forbids the orchestrator from making that final merge/PR/discard call itself — it must hand off to `finishing-a-development-branch`, keeping disposition decisions out of the execution skill's scope. Two distinct isolation layers, easy to conflate: per-track subagent isolation (`isolation="worktree"` on each dispatch, §3) is this skill's own mechanism; the orchestrator's *own* workspace isolation — needed because it directly merges track branches and writes state/roadmap files outside any track worktree — delegates to `using-git-worktrees` (§1), which is a no-op if the orchestrator is already isolated (e.g. dispatched inside a worktree by a higher-level pipeline).

#### `plan-review-eng` (skill)
- **Why needed:** A plan can look thorough while still having untested error paths, unjustified new dependencies, or a custom-rolled solution where the framework already has a built-in — an engineering-specific pass is needed to catch these before implementation burns time on them.
- **What it does:** Runs a Step-0 scope challenge (existing-code leverage, complexity check, built-in-vs-custom search check) then four confidence-calibrated review sections — architecture, code quality, test coverage (with a full ASCII coverage-diagram trace of every codepath and user flow), and performance — with a pre-emit verification gate requiring every finding to quote the motivating code line.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no persistent phase-review lens or confidence-calibration discipline for plan critique — findings in a normal review pass carry no confidence score and aren't required to quote the specific line that motivated them, so speculative findings get reported with the same weight as verified ones; this skill forces confidence to 4-5 (appendix-only) for anything unquotable.
- **When to use:** Frontmatter: "Use when reviewing a plan for architectural soundness and test completeness — directly or via the plan-reviewer agent with lens=eng."
- **Then what:** Produces a coverage diagram with GAP markers, a synthesized flat list of build-actionable implementation tasks (P1/P2/P3), and a verdict of APPROVE / APPROVE-WITH-CHANGES / REVISE.
- **Notes:** A Stage 7 lens — it reviews one phase's `PLAN.md`, not the project SDD. Project-level architecture strategy (the `SDD.md` + ADRs) is reviewed once at Stage 2 by `sdd-review-cto`; this lens assesses how a plan's choices fit that established design. Any missing regression test on modified existing behavior is an "iron rule" — always added to the plan as a critical requirement, never skipped, even when uncertain whether a change is technically a regression.

#### `plan-review-devex` (skill)
- **Why needed:** Plans for developer-facing products (APIs, CLIs, SDKs) get reviewed for correctness but rarely for whether a new developer can actually get to "hello world" in under 5 minutes — DX gaps like unclear error messages or a missing quick-start silently kill adoption after ship.
- **What it does:** Auto-detects the product type, builds an evidence-first investigation (developer persona, empathy narrative, competitive TTHW benchmark, magical-moment design, journey trace, first-time-developer confusion roleplay) before scoring 8 passes (getting started, API/CLI/SDK design, error messages, docs, upgrade path, tooling, community, DX measurement) each 0-10 against a gap-to-10 method.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standing Time-to-Hello-World benchmark or developer-persona discipline — asked to review a plan's onboarding it would offer generic advice; this skill requires every score to cite specific evidence from the Step-0 investigation (e.g. a named friction point from the journey trace), not a vibe-based number.
- **When to use:** Frontmatter: "Use when reviewing a plan for a developer-facing product (API, CLI, SDK, library, platform, docs) — directly or via the plan-reviewer agent with lens=devex."
- **Then what:** Produces a DX Scorecard, a Developer Journey Map with friction-point resolutions, and a verdict where TTHW in the "Red Flag" tier (>10 min) or any pass ≤2 is an automatic BLOCKER.
- **Notes:** Has an explicit "not applicable" gate — if the plan has no developer-facing surface at all, it reports so and auto-approves with completeness N/A rather than forcing a DX review onto a backend-only change.

#### `plan-review-goal-backward` (skill)
- **Why needed:** A plan can have every task filled in and still fail its own goal — e.g. an "auth endpoint" task exists but password hashing is missing — because task-completeness and goal-achievement are not the same thing, and nothing else in the review stack verifies the latter specifically.
- **What it does:** Adversarially assumes the plan is flawed until proven otherwise, and checks 12 verification dimensions — requirement coverage, task completeness, dependency correctness, artifact wiring ("key links"), scope sanity, verification-criteria derivation, decision compliance, silent scope-reduction detection, architectural tier compliance, verification feedback quality, cross-plan data contracts, and CLAUDE.md compliance — building an explicit requirement-to-task coverage map.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no dedicated "will this plan actually deliver the goal" verification pass distinct from a general code-quality review — without it, a plan with plausible-looking tasks and zero coverage on one requirement sails through unnoticed; this skill's Dimension 7b specifically hunts for scope-reduction language ("v1", "static for now", "stub") and always treats it as a BLOCKER, never a warning.
- **When to use:** Frontmatter: "Use when asking 'will this plan actually achieve its goal?' — directly or via the plan-reviewer agent with lens=goal-backward."
- **Then what:** Produces a coverage map, a classified blocker/warning list with fix hints, and either a VERIFICATION PASSED or ISSUES FOUND structured return recommending revision (or a phase/milestone split if scope reduction was detected).
- **Notes:** Explicitly static analysis only — it never runs the application; it verifies the plan describes what to build and how to prove it, not the running system itself.

#### `diagram` (skill)
- **Why needed:** Architecture and plan reviews repeatedly demand ASCII/diagram artifacts, but hand-drawn ASCII diagrams are hard to edit and don't survive iteration — teams need an editable, re-renderable diagram format, not a one-off image.
- **What it does:** Authors Mermaid from a plain-English request, renders it to SVG and PNG via mermaid-cli (or a native-rendering surface), optionally converts flowcharts to an editable `.excalidraw` scene, and always keeps the `.mmd` source as the single source of truth for later edits.
- **Why not vanilla Claude Code:** Vanilla Claude Code can write Mermaid syntax in a code block, but it has no standing convention for saving the source file, rendering it to multiple formats, or handling the Excalidraw round-trip — without this skill, "draw a diagram" produces a fenced code block that may or may not actually render, with no persistent editable artifact.
- **When to use:** Frontmatter: "Use when asked to 'draw a diagram', 'diagram this', 'make a flowchart', 'visualize the architecture', or turn mermaid into an image."
- **Then what:** Delivers an artifact set (`.mmd` + `.svg` + `.png`, optionally `.excalidraw`) with the PNG shown inline, and supports an iteration loop that edits the `.mmd` and re-renders rather than hand-editing the raster output.
- **Notes:** If no renderer is available, it explicitly refuses to "pretend a render happened" — it surfaces the exact install command and shows the raw Mermaid source instead.

#### `graphify` (skill)
- **Why needed:** Understanding a new or unfamiliar codebase (or a mixed corpus of docs/code/papers) by reading files one at a time misses cross-document relationships nobody would think to ask about directly — architecture comprehension needs a persistent, queryable structure, not repeated re-reading.
- **What it does:** Turns any folder of files (code, docs, papers, images, video) into a navigable knowledge graph via structural (AST) and semantic (LLM subagent) extraction, community detection, and an honest EXTRACTED/INFERRED/AMBIGUOUS audit trail, producing an interactive HTML graph, a GraphRAG-ready JSON, and a plain-language GRAPH_REPORT.md.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no persistent graph structure across sessions — relationships it infers while reading files evaporate at the end of the conversation; this skill's `graph.json` survives across sessions so a question asked weeks later can be answered by querying the graph instead of re-reading everything.
- **When to use:** Frontmatter: "Use when user asks any question about a codebase, documents, or project content - especially if graphify-out/ exists, treat the question as a /graphify query."
- **Then what:** Produces `graphify-out/graph.json`, `GRAPH_REPORT.md`, and `graph.html`, then offers to trace the single most interesting cross-community question the graph surfaced.
- **Notes:** Directly consumed by the `planner`, `phase-researcher`, and (for `arch`/`concerns` focus only) `codebase-mapper` agents, which query `graphify` when a graph already exists — for `codebase-mapper`, its community detection and hotspot analysis (`god_nodes`/`surprising_connections`) map onto ARCHITECTURE.md's Layers/Key Abstractions and CONCERNS.md's Fragile Areas, so it's used as a starting point before fresh exploration rather than re-derived from scratch.

#### `planner` (agent)
- **Why needed:** Turning a phase goal into an executable plan requires simultaneously tracking task dependencies, context-budget limits per task, goal-backward verification criteria, and user decision fidelity — doing this ad hoc in a chat produces plans that are too big, miss requirements, or silently water down locked decisions.
- **What it does:** Decomposes a phase into 2-3-task plans grouped into dependency-ordered waves, derives goal-backward must-haves (observable truths → artifacts → wiring → key links), enforces a strict prohibition on scope-reduction language ("v1", "static for now"), and produces `PLAN.md` files that Claude executors can implement without interpretation. **Authors each plan's tasks via the `writing-plans` skill** (the canonical `<task>` format lives there); this agent owns only the pipeline wrapping around it — multi-plan wave/track decomposition, dependency graph, frontmatter, the `## Parallel Execution Map`, file naming, native validation, and git.
- **Why not vanilla Claude Code:** Vanilla Claude Code writing a plan in a normal session has no context-budget discipline (each plan should complete within ~50% context) or wave-dependency-graph logic — it also has no built-in mechanism to detect when a plan silently simplifies a user's locked decision instead of delivering it fully, which this agent's `scope_reduction_prohibition` explicitly hunts for.
- **When to use:** Dispatched by the orchestrator/pipeline for standard phase planning, gap closure (from verification failures), revision (from checker feedback), or replanning with cross-AI review feedback.
- **Then what:** Writes `{padded_phase}-{NN}-PLAN.md` files with full frontmatter (wave, depends_on, files_modified, must_haves), updates ROADMAP.md placeholders, and commits — then reports wave structure and next steps (`/gsd:execute-phase`).
- **Notes:** Has "no authority to judge a feature as too difficult" — it may only split or flag work for three narrow reasons (context cost >50%, missing information, dependency conflict), never because a feature seems complex.

#### `roadmapper` (agent)
- **Why needed:** Turning a requirements list into a phase structure by hand risks arbitrary "Setup → Core → Features → Polish" templates that don't reflect the actual dependency structure of the work, and risks orphaned requirements nobody notices until much later.
- **What it does:** Derives phases from requirement categories and their dependencies (not an imposed template), applies goal-backward reasoning at the phase level (state the goal → derive 2-5 observable user-facing truths → cross-check against requirements), and validates 100% requirement coverage before writing `ROADMAP.md` and initializing `STATE.md`.
- **Why not vanilla Claude Code:** Vanilla Claude Code proposing a roadmap has no coverage-validation step forcing every requirement to map to exactly one phase — it also defaults toward generic project-management structure (sprints, ceremonies) that this agent's philosophy explicitly forbids as "corporate PM theater" for a solo-developer + Claude workflow.
- **When to use:** Dispatched by the orchestrator/pipeline during unified project initialization, after requirements and (optionally) research are available.
- **Then what:** Writes `ROADMAP.md` (both a summary checklist and detailed `### Phase X:` sections parsed by downstream tools) and `STATE.md`, presents a draft for user approval, and reports 100%-coverage confirmation.
- **Notes:** Explicitly rejects horizontal-layer phasing (all models → all APIs → all UI) in favor of vertical slices that each deliver one complete, independently verifiable capability.

#### `phase-researcher` (agent)
- **Why needed:** Planning a phase without first investigating its specific technical domain risks the planner hand-rolling a solution to a problem the ecosystem already solved, or missing a well-known pitfall that causes a rewrite later.
- **What it does:** Investigates the phase's technical domain (standard stack, architecture patterns, don't-hand-roll list, common pitfalls) via a strict Context7 → official docs → WebSearch verification hierarchy, tags every claim with provenance (`[VERIFIED]`/`[CITED]`/`[ASSUMED]`), runs a package-legitimacy gate against supply-chain risk, and writes a single `RESEARCH.md` the planner consumes.
- **Why not vanilla Claude Code:** Vanilla Claude Code treats its own training knowledge as fact rather than a stale hypothesis — this agent explicitly requires verification before asserting library capabilities and forces every `[ASSUMED]` claim into an Assumptions Log the planner must gate behind a human-verify checkpoint before install, something a normal session has no mechanism to enforce.
- **When to use:** Dispatched by the orchestrator/pipeline, either integrated into `/gsd:plan-phase` or standalone via `--research-phase <N>`.
- **Then what:** Writes `{phase}-RESEARCH.md` with a prescriptive (not exploratory) Standard Stack, Package Legitimacy Audit, Architecture Patterns, and (when relevant) a Runtime State Inventory for rename/migration phases — the planner then builds tasks directly against it.
- **Notes:** A package name discovered via WebSearch or training data must be tagged `[ASSUMED]` even if `npm view` confirms it exists on the registry — registry existence alone doesn't prove legitimacy, since a slopsquatted package also passes that check.

#### `plan-reviewer` (agent)
- **Why needed:** Five distinct plan-review lenses (CEO, engineering, design, DevEx, goal-backward) each have deep, incompatible methodologies — running them all inside one context would blur their distinct rigor; something needs to apply exactly one lens per dispatch and return a uniform, comparable report.
- **What it does:** Reads the plan file plus provided context, loads the specific `plan-review-<lens>/SKILL.md` file for its assigned lens, executes that skill at full depth (every section/pass/dimension evaluated, "no findings" recorded not skipped), and writes a report in a fixed contract (Findings table, Lens Report, Completeness score, Verdict).
- **Why not vanilla Claude Code:** Vanilla Claude Code asked to review a plan through "the CEO lens" and "the engineering lens" in one pass will blend the two perspectives and lose the calibrated rigor each skill defines — this agent enforces single-lens isolation per dispatch and a uniform severity/verdict contract across all five lenses so their outputs can be consolidated apples-to-apples.
- **When to use:** Dispatched once per selected lens (typically in parallel, one per lens) by the `plan-review` command or any orchestrator wanting a structured plan critique.
- **Then what:** Writes `<plan-dir>/reviews/<plan-basename>.<lens>-review.md` and returns a compact summary (report path, verdict, completeness score, severity counts, top 3 findings) under 20 lines.
- **Notes:** If the assigned lens's skill file can't be found, it stops and reports the failure rather than improvising the lens from memory — the skill file is treated as the single source of truth for HOW to review.

#### `pattern-mapper` (agent)
- **Why needed:** Executors implementing new files from scratch tend to invent their own conventions rather than matching what the codebase already does — without an explicit analog mapping, a plan's tasks read "follow the auth pattern" with no concrete reference, forcing the executor to hunt through the codebase itself.
- **What it does:** Classifies every file the phase will create/modify by role and data-flow type, searches the codebase for the closest existing analog per file, reads each analog and extracts concrete code excerpts (imports, auth pattern, core pattern, error handling), and writes a `PATTERNS.md` mapping new files to specific line-numbered examples to copy.
- **Why not vanilla Claude Code:** Vanilla Claude Code writing a plan task will describe a pattern in the abstract ("follow the auth pattern") rather than citing the exact file and line range to copy from — this agent's downstream_consumer contract explicitly requires "concrete, not abstract" excerpts with file paths and line numbers so the planner can paste them directly into task actions.
- **When to use:** Dispatched by the orchestrator/pipeline between the research and planning steps, read-only against the codebase.
- **Then what:** Writes `PATTERNS.md` to the phase directory with per-file pattern assignments, shared cross-cutting patterns (auth, error handling), and a "no analog found" list — the planner then references these analogs directly in task actions.
- **Notes:** Strictly read-only for source code — `PATTERNS.md` is the only file it writes; it never modifies application code.

#### `codebase-mapper` (agent)
- **Why needed:** Every phase plan needs to know existing conventions, directory structure, and known technical debt to avoid contradicting them or repeating past mistakes — re-deriving this from scratch each time wastes context and produces inconsistent guidance.
- **What it does:** Explores the codebase for one of four focus areas (tech/arch/quality/concerns) and writes structured documents directly to `.planning/codebase/` (STACK/INTEGRATIONS, ARCHITECTURE/STRUCTURE, CONVENTIONS/TESTING, or CONCERNS), always including concrete file paths and current-state-only descriptions.
- **Why not vanilla Claude Code:** Vanilla Claude Code re-explores the codebase from scratch on every phase's planning pass since it has no standing "here's what this project looks like" reference document — this agent writes documents directly to disk specifically to reduce context transfer to the orchestrator and to give the planner/executor a persistent reference they can load selectively by phase type.
- **When to use:** Dispatched by the orchestrator/pipeline with a specific focus area, typically ahead of planning or after significant codebase drift.
- **Then what:** Writes the focus-specific UPPERCASE.md document(s) to `.planning/codebase/` and returns only a brief confirmation (~10 lines) — never the document contents — so context stays cheap.
- **Notes:** Supports an optional `--paths` scope hint used by the incremental-remap post-execute codebase-drift gate, restricting exploration to specific repo-relative subtrees instead of re-scanning the whole repository. For `arch`/`concerns` focus, checks for an existing `graphify` graph first and queries it before fresh exploration — its already-clustered communities and hotspot analysis overlap directly with what those two focuses would otherwise re-derive by hand.

#### `gate-plan-review` (agent)
- **Why needed:** A written sprint plan's Parallel Execution Map can silently misdeclare a track's model/effort tier relative to its actual complexity signals, or drift from documented architectural decisions — someone independent of the planner needs to catch this before Wave 1 ever dispatches.
- **What it does:** Runs a deterministic complexity-score check (scripted or manual fallback, against the canonical signals in `references/complexity-signals.md`) validating every track's declared Model/Effort against its complexity signals, then selects a review engine per `references/independent-review.md` (default Gemini, falling back to Codex then Claude) to review the plan against the SDD/ADRs for alignment, ADR gaps, scope coverage, structural soundness, vertical-slice compliance, and signal honesty — paths only, never inline content — producing a schema-compliant JSON summary.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no independent, deterministic gate on a plan's complexity self-declarations — a planner grading its own tracks as "low effort" has no external check; this agent runs a separate scorer (or a manual judgment fallback) specifically to catch under-declared signals that would game the deterministic model-selection logic.
- **When to use:** Spawned after a sprint plan (with its Parallel Execution Map) has been written, before Wave 1 dispatch.
- **Then what:** Writes `<plan_path>.review.md` (full prose) and `<plan_path>.review-summary.json` (schema-compliant, including which engine ran), then returns only the JSON to the orchestrator — `gate_passed: true` unlocks Wave 1 dispatch.
- **Notes:** Complexity blockers from the deterministic scorer can never be waived with an inline "won't fix" justification the way the review engine's HIGH findings sometimes can — they must be mechanically corrected and the scorer re-run. No longer hardcoded to Gemini — see `references/independent-review.md` for the engine registry.

#### `gate-reverse-engineer` (agent)
- **Why needed:** An existing, undocumented codebase has no SDD or PRD to plan against — before any architectural planning can happen on a legacy system, someone has to empirically recover what the system actually does and why, distinct from what anyone intended it to do.
- **What it does:** Scans an existing codebase's entry points, config, and data schemas, drafts a "Legacy SDD" (components, data flow, responsibilities) and "Legacy PRD" (inferred user requirements and business logic from code and tests), generates retrospective ADRs for existing technology choices, and flags any contradiction between its empirical findings and existing documentation as an "Inconsistency."
- **Why not vanilla Claude Code:** Vanilla Claude Code exploring a legacy codebase has no standing process for converting empirical findings into the same document types (SDD/PRD/ADR) the rest of the planning pipeline expects — without this agent, "reverse engineering" produces an ad hoc summary that downstream planning tools can't consume the way they consume a real SDD/PRD.
- **When to use:** Dispatched to analyze an existing, poorly-documented codebase and recover its design intent, technical stack, and business requirements.
- **Then what:** Writes `docs/architecture/SDD.md`, `docs/requirements/PRD.md`, and ADRs to `docs/architecture/ADRs/`; hidden technical debt or security risks it finds are explicitly highlighted for the subsequent `code-review-gate` round-mode pass.
- **Notes:** Explicitly follows the `spec-miner` skill's Arch-Hat/QA-Hat methodology for the deep-scan itself — it's a downstream consumer of that skill's extraction technique, applied specifically toward recovering architecture-driven artifacts (SDD/PRD/ADRs) rather than a generic reverse-engineered spec.

#### `plan-review` (command)
- **Why needed:** Running all five plan-review lenses one at a time in the same session is slow and risks cross-contaminating each lens's calibrated judgment — someone needs a single entry point that fans out to all (or selected) lenses in parallel and consolidates the result.
- **What it does:** Parses a plan file path plus optional lens names (defaulting to all 5: ceo/eng/design/devex/goal-backward), dispatches the `plan-reviewer` agent once per selected lens in parallel, and expects one report per lens consolidated into a single set of findings for the plan.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standing command that fans a single plan out across five independently-calibrated review methodologies in parallel — without this command, a user would have to manually invoke each `plan-review-<lens>` skill in sequence, losing both the parallelism and the uniform-report consolidation the `plan-reviewer` agent provides.
- **When to use:** Invoked directly by the user (or an orchestrator) on a plan file, optionally scoped to specific lenses.
- **Then what:** Produces N parallel lens reports (each following the `plan-reviewer` agent's uniform contract), consolidated into a single findings set for the plan under review.
- **Notes:** Is a thin dispatcher — all actual review logic lives in the `plan-reviewer` agent and the five `plan-review-*` skills; the command itself contains no review methodology.

## Role: UX / Design

#### `design-consultation` (skill)
- **Why needed:** UI work started with no design system produces incoherent, ad hoc visual decisions made file-by-file — someone has to establish typography, color, layout, spacing, and motion as one coherent system before any screen gets built, or every screen ends up looking like it came from a different team.
- **What it does:** Gathers product context (including a "memorable thing" forcing question), optionally researches comparable products across three layers (tried-and-true, current trend, first-principles), proposes a complete coherent design package with explicit safe-choices vs. deliberate-risk framing, generates a self-contained HTML preview with font/color/mockup sections, and writes the approved system to `DESIGN.md`. Includes a "Variant shotgun" mode for generating and comparing several competing full directions side-by-side.
- **Why not vanilla Claude Code:** Vanilla Claude Code asked to "make it look nice" has no standing anti-convergence discipline — it will reach for the same overused fonts (Inter, Space Grotesk) and AI-slop patterns (purple gradients, 3-column icon grids) every time; this skill maintains an explicit font blacklist/overused list and a self-gate ("would a human designer be embarrassed to put their name on this?") before presenting any preview.
- **When to use:** Frontmatter: "Use when asked for a 'design system', 'brand guidelines', 'create DESIGN.md', 'explore design directions', or when starting a new project's UI with no existing design system."
- **Then what:** Writes `DESIGN.md` (and updates CLAUDE.md to require reading it before any visual decision) as the project's design source of truth, then suggests running `design-html` to see it as a working page.
- **Notes:** `design-html`, `design-handoff`, and the `ui-researcher` agent's UI-SPEC.md all treat this skill's `DESIGN.md` output as an upstream authority that takes priority over their own inferred values.

#### `design-handoff` (skill)
- **Why needed:** A `DESIGN.md` document written for human designers isn't directly actionable for an implementing agent or developer — someone has to translate mood statements and hex tables into copy-paste-ready component prompts without losing fidelity in either direction.
- **What it does:** Extracts and summarizes DESIGN.md (and UI-SPEC.md if present) across nine standard sections (theme, color, typography, components, layout, elevation, responsive behavior, do's/don'ts, agent prompt guide), then synthesizes bullet-point instructions with quick-reference tables and 3-5 ready-to-paste component-build prompts, flagging any gap the source didn't specify.
- **Why not vanilla Claude Code:** Vanilla Claude Code handed a DESIGN.md and asked to "implement it" will paraphrase loosely and may silently invent values for anything underspecified — this skill's explicit rule is "recommend defaults only in the clearly-marked Gaps section," never silently filling gaps, and never modifying values without an explicit request.
- **When to use:** Frontmatter: "Use when a design system exists and someone (or some agent) needs to implement UI that faithfully matches it — 'hand off the design', 'prep build instructions from DESIGN.md', 'write prompts for the UI agent', 'replicate this look'."
- **Then what:** Saves `.claude/design/instructions-<name>.md` (or `docs/design-handoff.md`) and reports counts of colors/typography rules/components/prompts extracted plus any flagged gaps; suggests handing the file to `design-html` or a frontend developer agent.
- **Notes:** Stops and suggests running `design-consultation` first if no `DESIGN.md` exists at all — it refuses to invent a design system rather than translate from nothing.

#### `design-html` (skill)
- **Why needed:** An approved mockup or design plan still has to become real, responsive, accessible HTML/CSS with real content — without a disciplined finalization step, "turn this into code" produces layouts that break on resize, use lorem ipsum, or ignore dark mode and accessibility entirely.
- **What it does:** Detects the input context (approved mockup / plan-driven / freeform / evolve), classifies the layout strategy per design type, generates a single self-contained production-quality HTML file (CSS custom properties, semantic HTML5, ARIA, `prefers-color-scheme`, `prefers-reduced-motion`, real content never lorem ipsum), serves a live preview, and runs a surgical-edit refinement loop capped at 10 iterations.
- **Why not vanilla Claude Code:** Vanilla Claude Code generating "a landing page" defaults toward a documented AI-slop blacklist (purple gradients, generic 3-column features, cookie-cutter hero sections) that this skill explicitly forbids — it also has no standing rule against regenerating the whole file on every refinement request, which would clobber a user's manual edits; this skill mandates targeted Edit-tool changes only.
- **When to use:** Frontmatter: "Use when asked to 'finalize this design', 'turn this into HTML', 'build me a page', 'implement this design', 'code the mockup', or after a design-consultation session."
- **Then what:** Saves the finalized HTML (or framework component) plus `finalized.json` metadata, and offers to extract a `DESIGN.md` from the generated tokens if none exists yet.
- **Notes:** Explicit fidelity rule — when an approved mockup exists, match it exactly even at the cost of code elegance (e.g. a literal `width: 312px`); cleanup happens later during component extraction, not during finalization.

#### `plan-review-design` (skill)
- **Why needed:** A plan can describe backend logic thoroughly while leaving "what does the user actually see" almost entirely unspecified — without a design-specific plan review, missing empty states, unresolved responsive behavior, and generic AI-slop UI descriptions ship straight into implementation.
- **What it does:** Runs 7 scored passes (information architecture, interaction-state coverage, user journey/emotional arc, AI-slop risk with hard-rejection criteria and a pattern blacklist, design-system alignment, responsive/accessibility, unresolved decisions) each rated 0-10 with a "rate → gap → fix" method that writes the missing design decisions directly back into the plan's findings.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no persistent AI-slop blacklist or hard-rejection criteria to check a plan's described UI against — it will happily accept "cards with icons" or "clean, modern UI" as sufficient design detail; this skill's Pass 4 explicitly classifies the plan as marketing/app/hybrid and applies a different rule set for each, rejecting vague descriptions outright.
- **When to use:** Frontmatter: "Use when reviewing a plan with any UI scope — directly or via the plan-reviewer agent with lens=design."
- **Then what:** Produces per-pass before/after scores, an interaction-state table, a user-journey storyboard, and an Unresolved Decisions table with consequences — verdict REVISE if any hard-rejection criterion triggers or a core screen has no specified states.
- **Notes:** Has an explicit UI-scope gate — if the plan touches no UI at all, it reports "not applicable," verdict APPROVE, completeness N/A, rather than forcing a design pass onto a backend-only change.

#### `devex-review` (skill)
- **Why needed:** A DX plan review can only judge intent — someone still has to actually walk the real getting-started flow, run the real CLI `--help`, and trigger real errors to know whether the shipped experience matches what was planned, because plans and reality diverge.
- **What it does:** Actually tests the live developer experience — times Time-to-Hello-World against tiered benchmarks, runs CLI commands and evaluates `--help` output, triggers real error scenarios and grades them against a three-tier quality model (Elm/Rust/Stripe-style), audits docs/upgrade-path/dev-environment/community, and runs a separate inner-loop workflow-optimization pass (build times, HMR latency, test suite runtime).
- **Why not vanilla Claude Code:** Vanilla Claude Code reviewing DX from documentation alone is reading about the experience, not testing it — this skill explicitly marks every score's evidence source (TESTED / PARTIAL / INFERRED) and forbids treating an inferred score as equivalent to one backed by an actual command run or screenshot.
- **When to use:** Frontmatter: "Use when asked to 'test the DX', 'DX audit', 'developer experience review', 'try the onboarding', or after shipping a developer-facing feature."
- **Then what:** Produces a DX Scorecard with evidence-labeled scores per dimension plus a measured TTHW, and recommends specific fixes ordered by adoption-impact-over-effort, suggesting a re-run after fixes to verify improvement.
- **Notes:** Distinct from `plan-review-devex` — that skill reviews a plan's DX intent before implementation; this skill tests the actual shipped product, live, after the fact.

#### `ui-researcher` (agent)
- **Why needed:** Frontend phases need a concrete, pre-agreed design contract (exact spacing, exact type scale, exact copy) before planning starts — without one, the planner and executor each invent their own visual decisions independently, guaranteeing inconsistency.
- **What it does:** Reads upstream CONTEXT/RESEARCH/REQUIREMENTS to pre-populate what's already decided, detects existing design-system state (shadcn, Tailwind tokens, existing components) via codebase scan, runs a shadcn-initialization gate, asks only the design-contract questions upstream artifacts didn't already answer (spacing, typography, 60/30/10 color, copywriting), and runs a registry-safety vetting gate on any third-party shadcn blocks before including them.
- **Why not vanilla Claude Code:** Vanilla Claude Code building a frontend phase has no standing design-contract format — spacing and type choices get made ad hoc per file; this agent enforces hard constraints (max 4 font sizes, max 2 weights, spacing multiples of 4, accent color reserved for a specific named list of elements) that a normal session has no mechanism to check itself against.
- **When to use:** Dispatched by the orchestrator/pipeline for frontend phases needing a `UI-SPEC.md` design contract.
- **Then what:** Writes `UI-SPEC.md` (status: draft) to the phase directory — the `ui-checker` agent then validates it before the planner and executor treat it as visual source of truth.
- **Notes:** If a third-party registry block is declared but the user refuses the safety-vetting gate, the agent returns UI-SPEC BLOCKED rather than silently including an unvetted component — a hard dependency on the vetting step actually being completed, not just intended.

#### `ui-checker` (agent)
- **Why needed:** A UI-SPEC.md can have every section technically filled in and still produce design debt — an accent color "reserved for all interactive elements" or a generic "Submit" CTA label passes a superficial completeness check but defeats the whole purpose of having a contract.
- **What it does:** Validates the design contract against 6 quality dimensions (copywriting, visuals, color, typography, spacing, registry safety) with explicit BLOCK/FLAG criteria per dimension — e.g. more than 4 font sizes is an automatic BLOCK, a registry safety-gate column showing only "view + diff required" (intent, not evidence) is a BLOCK.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standing rubric distinguishing "the section is filled in" from "the section actually prevents design debt" — this agent's dimension criteria specifically target contract text that looks complete but is functionally useless (e.g. an accent color list that includes everything), which a generic review would pass.
- **When to use:** Dispatched by the orchestrator/pipeline after the `ui-researcher` creates UI-SPEC.md, or for re-verification after a revision.
- **Then what:** Returns APPROVED (all dimensions PASS/FLAG) or BLOCKED (any dimension BLOCK) with exact fix descriptions; planning cannot proceed while BLOCKED.
- **Notes:** Strictly read-only — it never edits UI-SPEC.md itself; the `ui-researcher` must apply the fixes and the checker re-verifies.

#### `ui-auditor` (agent)
- **Why needed:** Once a frontend phase is implemented, someone has to check whether the actual rendered/coded UI still matches the UI-SPEC.md contract (or abstract standards if none exists) — implementation frequently drifts from the design contract in ways nobody notices until a user complains.
- **What it does:** Captures screenshots (browser tooling preferred, Playwright CLI fallback, code-only audit if no dev server), scores 6 pillars 1-4 each (copywriting, visuals, color, typography, spacing, experience design) via grep-based evidence plus screenshot comparison against UI-SPEC.md, runs a post-scoring registry-safety audit on any shadcn third-party blocks, and writes `UI-REVIEW.md` with the top 3 priority fixes.
- **Why not vanilla Claude Code:** Vanilla Claude Code asked "does the UI match the design" has no adversarial default — it tends to average scores upward and accept "the component exists" as sufficient evidence; this agent's explicit adversarial stance assumes deviation until code/screenshot evidence proves otherwise, and requires every scored pillar to cite at least one specific finding.
- **When to use:** Dispatched by the orchestrator/pipeline retroactively, after a frontend phase's implementation is complete.
- **Then what:** Writes `UI-REVIEW.md` (pillar scores out of 24, top 3 priority fixes, detailed findings with file:line references) to the phase directory.
- **Notes:** Runs a mandatory `.gitignore` gate before any screenshot capture, specifically to prevent binary screenshot files from ever reaching git history even if the user later runs `git add .`.

#### `design-reviewer` (agent)
- **Why needed:** Static code or plan review can't catch how a live site actually feels to use — sluggish transitions, inconsistent spacing across pages, or AI-slop visual patterns only show up when someone drives the real, rendered app and reacts to it as a user would.
- **What it does:** Drives a live browser through documented user flows adopting a "slightly frustrated real user" persona, forms a First Impression, extracts the actually-rendered design system via JS introspection, runs a 10-category audit checklist (hierarchy, typography, color, spacing, interaction states, responsive, motion, content, AI-slop detection, performance) with a trunk test per page, then iteratively fixes findings in source with atomic commits and before/after screenshot verification, self-regulating against a revert-risk heuristic.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standing mechanism to actually drive a browser, extract the live rendered design system via JS, and then fix what it finds with atomic, reversible commits — a normal review pass only reads source and can't verify the rendered result matches, nor does it have this agent's self-regulation stop-and-ask gate when accumulated fix risk exceeds 20%.
- **When to use:** Dispatched by the orchestrator/pipeline for a live, browser-driven design audit of a running app (full/quick/deep/diff-aware/regression modes).
- **Then what:** Writes a design-audit report with before/after screenshots and commit SHAs per fix, a design-score and AI-slop-score delta, and updates `TODOS.md` for deferred findings.
- **Notes:** Requires a clean git working tree before starting (since the fix loop makes real commits), and reverts immediately on any regression detected during re-testing rather than leaving a broken fix in place.

#### `accessibility-tester` (agent)
- **Why needed:** Visual design review covers whether an interface looks intentional, but it doesn't verify whether it's actually usable by keyboard-only users, screen-reader users, or people needing higher contrast — that requires a dedicated WCAG-conformance pass most design reviews don't run.
- **What it does:** Establishes the compliance target (default WCAG 2.1 AA), reviews existing accessibility implementation and known violations, checks keyboard navigation/focus management, screen-reader announcement order and ARIA correctness, visual contrast/zoom/motion controls, form label associations, and cognitive/mobile accessibility (touch targets, gesture alternatives), combining automated scanners with manual verification.
- **Why not vanilla Claude Code:** Vanilla Claude Code has no standing WCAG success-criteria checklist or assistive-technology coverage model — a generic review might notice obvious contrast issues but has no systematic pass over keyboard traps, ARIA landmark correctness, or screen-reader announcement order; this agent treats automated scanner results as "a floor, not proof," always pairing them with manual verification.
- **When to use:** Frontmatter: "Use for accessibility testing, WCAG compliance verification, and assessment of assistive-technology support."
- **Then what:** Reports findings ordered by severity (critical/serious/moderate/minor) each with the WCAG criterion, affected location, and a specific fix, prioritizing quick wins before structural fixes; does not fix code unless explicitly asked.
- **Notes:** Complements `design-reviewer` and `ui-auditor` — those cover visual/interaction quality broadly, while this agent is the dedicated pass for assistive-technology and WCAG conformance specifically.
